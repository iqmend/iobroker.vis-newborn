# Session-Log 2026-05-05 — `iobroker.vis-newborn`

Vollständige Entwicklung des Adapters in einer Session: vom ersten Push bis v0.1.11. Hier die Chronologie mit allen Entscheidungen, Sackgassen und Lessons.

## Ausgangspunkt

- Vorgänger-Repo `iobroker.vis-newborn-toggle` v0.1.7 lief produktiv (Slide-Toggle, 1- oder 2-DP-KNX, 3 Größen)
- Neues Dimmer-Widget gewünscht: HomeKit-Style Kachel, Tap = An/Aus, Long-Press öffnet Vertikal-Slider-Popup, KNX-tauglich (4 DPs)
- Plus: User wollte später beide Widgets (und folgende) unter einem Adapter `iobroker.vis-newborn` zusammenführen statt separater Repos

## Fertige Widgets (Stand v0.1.11)

| Widget | Template-ID | Beschreibung |
|---|---|---|
| Newborn Toggle (Slide) | `tplNewbornToggle` | Slide-Toggle, 1- oder 2-DP-KNX-Modus, optionaler Status-Invert |
| Newborn Dimmer (Tile) | `tplNewbornDimmer` | HomeKit-Style Kachel, Tap = An/Aus, Long-Press öffnet Vertikal-Slider-Popup, 4 DPs |
| Newborn Licht (Tile) | `tplNewbornLight` | Wie Dimmer, ohne %-Anzeige; KNX-Modus per Auto-Detect über `oid_status` |
| Newborn Jalousie (Tile) | `tplNewbornBlinds` | Hochformat-Kachel mit Jal+Lam Auf/Ab + Stop + Long-Press-Popup mit zwei Slidern, 7 OIDs (inkl. Fenster-Badge) |

Alle Widgets erscheinen unter Palette-Sektion „newborn". Alle haben:
- 3 Default-Größen (`size`-Dropdown: small/medium/large)
- Optionale Custom-Maße (`width`/`height`-Felder mit Min-Clamp auf small)
- Auto-Subscription via `^oid`-Regex an `widget.data`-Attributnamen

## Wichtigste Lessons (Kurzfassung)

1. **Filename-Stem matcht Adapter-Suffix** — sonst löscht VIS die Datei. Alle Widgets gehören in **eine** `widgets/<stem>.html`.
2. **jQuery-Events proxieren `touches` nicht** — `e.originalEvent.touches` lesen, oder `getXY()`-Helper.
3. **Bei Debug-Spirale revertieren statt patchen** — siehe v0.1.3 + v0.1.5 als abgelöste Versionen.
4. **Source lesen vor Diagnose** — die echte Ursache von 0.1.0 → 0.1.1 (Stem-Trap) stand in `iobroker.vis/lib/install.js`. Diagnose-Checklisten haben das nicht zutage gefördert; eine Source-Recherche schon.
5. **Bedingte Editor-Sichtbarkeit gibt es in vis 1.5.x nicht** — Workaround: Auto-Detect (z. B. `oid_status` gesetzt = KNX-Modus) oder zwei separate Widget-Templates.
6. **`<% %>`-Blöcke in EJS-Templates** für Editor-Config-Logik OK, aber **keine geschweiften Klammern** verwenden — Single-Line `if (cond) { x = y; }` ist eine VIS-`{...}`-Substitution-Falle. Klammerlos: `if (cond) x = y;`.
7. **CSS `touch-action: manipulation` + Light-Theme nur am Wrap, nicht doppelt am Tile** — die doppelte Anwendung in v0.1.3 hat die Klick-Handler zerschossen.
8. **Custom Width/Height per `<% %>`-Block + Inline-Style mit `!important`** — Min-Clamp auf jeweilige `small`-Größe pro Widget.
9. **Long-Press-Logik mit jQuery-Events ist auf Mobile nie 100 % perfekt** — Synth-Mouse-Race und Selection-Callout sind Restprobleme. Pointer-Events-API wäre die saubere Lösung, hat aber in vis 1.5.6 zu Render-Bugs geführt (v0.1.5).

Detail-Lessons in [vis-widget-development.md](vis-widget-development.md).

## Offene Restprobleme (akzeptiert)

- **Long-Press auf Mobile zeigt kurz Selection-Highlight** vor dem Popup — kosmetisch, Funktion intakt
- **1-von-10 Toggle+Popup-Race auf Mobile** beim Long-Press — selten, akzeptiert
- **Pointer-Events-Migration aufgeschoben** bis VIS-1.5.6-Render-Pipeline vollständig durchverstanden ist (`extractBinding` / `formatBinding` / `new Function` in `vis.js`)
